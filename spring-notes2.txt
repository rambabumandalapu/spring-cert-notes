Question 20
Using the Spring AOP framework, what is the visibility of the method matches by the following join point?

@Pointcut("execution(* *(..))") private void anyOperation() {};

1. All methods, whereas their visibility
2. All methods, except private method
3. Protected and public methods
4. ✅Public methods
Correct answer is the number 4.  

Due to the proxy-based nature of Spring's AOP framework, protected methods are by definition not intercepted, neither for JDK proxies nor for CGLIB proxies. As a consequence, any given pointcut will be matched against public methods only!  

To intercept private and protected methods, AspectJ weaving should be used instead of the Spring’s proxy-based AOP framework.

⚖️ Sequential Flow Visualization
Let’s imagine a method call with all advice types applied:

@Before → executes first.

@Around (before part) → executes next.

Target method executes.

If method returns normally:

@AfterReturning executes.

@After executes.

@Around (after part) executes.

If method throws exception:

@AfterThrowing executes.

@After executes.

@Around (after part) executes.

🔎 Flow Diagram (Textual)
Code
@Before
   ↓
@Around (before)
   ↓
Target Method
   ├── Normal return → @AfterReturning → @After → @Around (after)
   └── Exception     → @AfterThrowing → @After → @Around (after)
🎯 Key Takeaway
Before always runs first.

Around wraps everything.

AfterReturning runs only on success.

AfterThrowing runs only on failure.

After always runs at the end (like finally).

Here’s a **cheat sheet of Spring MVC exception handling mechanisms** — concise, exam‑ready, and easy to compare:

---

## 🧩 Exception Handling Cheat Sheet

| Mechanism | Where to Use | Trigger | Typical Usage |
|-----------|--------------|---------|---------------|
| **`@ResponseStatus`** | On exception classes or controller methods | Sets a fixed HTTP status when exception is thrown or method returns | Map custom exceptions to HTTP codes (e.g., `404 Not Found`) |
| **`ResponseStatusException`** | Thrown directly in controller/service | Runtime exception carrying HTTP status + reason | Dynamic error responses (`throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Invalid input")`) |
| **`@ExceptionHandler`** | Inside a controller class | Handles specific exceptions thrown by methods in that controller | Return custom response bodies or views for errors |
| **`@ControllerAdvice`** | Global, cross‑cutting | Handles exceptions across all controllers | Centralized error handling, logging, consistent error JSON |
| **`ExceptionHandlerExceptionResolver`** | Framework resolver | Delegates to `@ExceptionHandler` methods | Internal Spring mechanism to wire exception handlers |
| **`ResponseStatusExceptionResolver`** | Framework resolver | Detects `@ResponseStatus` annotations | Maps annotated exceptions to HTTP status codes |

---

## 🔎 Quick Examples

### 1. `@ResponseStatus` on Exception
```java
@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException { }
```

### 2. Throwing `ResponseStatusException`
```java
@GetMapping("/item/{id}")
public Item getItem(@PathVariable String id) {
    throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Invalid ID");
}
```

### 3. Local `@ExceptionHandler`
```java
@ExceptionHandler(ResourceNotFoundException.class)
public ResponseEntity<String> handleNotFound(ResourceNotFoundException ex) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND).body("Item not found");
}
```

### 4. Global `@ControllerAdvice`
```java
@ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)
    public ResponseEntity<String> handleAll(Exception ex) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body("Something went wrong");
    }
}
```

---

## 🎯 Takeaways
- **Use `@ResponseStatus`** for static mappings (exception → status).  
- **Use `ResponseStatusException`** for dynamic, runtime decisions.  
- **Use `@ExceptionHandler`** for controller‑specific handling.  
- **Use `@ControllerAdvice`** for global, centralized handling.  
- **Resolvers (`ExceptionHandlerExceptionResolver`, `ResponseStatusExceptionResolver`)** are Spring’s internal engines that make these annotations work.
